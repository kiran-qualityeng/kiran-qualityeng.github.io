Test Cases

Input Validation — Robust Sanitization & Length Handling (EC-IV)

Priority: Medium / High
Focus: XSS, SQLi, Length Limits
  1. EC-IV-001 — Special Character Handling
    1. Preconditions: Forms and APIs accepting user text fields are available.
    2. Steps:
      1. Submit input: John<script>alert('xss')</script>Doe in name/comment fields.
      2. Submit input: test'; DROP TABLE users;-- via form and direct API payload.
      3. Submit a normal input string and compare behaviour.
    3. Expected: Inputs are sanitized or rejected; no script execution in UI; server rejects/validates and encodes on output; logs show validation errors (not stack traces).
    4. Verification: Inspect DOM for injected scripts, review network payloads, and confirm DB stored values are encoded or rejected.
  2. EC-IV-002 — Extremely Long Inputs
    1. Preconditions: UI and API have defined max length policies.
    2. Steps:
      1. Enter 500 characters in name field; 10,000 in description; 1000 in search; submit forms and API calls.
      2. Measure response latency and observe behavior with very long Unicode/multibyte strings.
    3. Expected: UI truncates or prompts error; API validates payload length and returns 4xx with clear message; no backend slowdowns or crashes; DB stores allowed lengths only.
    4. Verification: Monitor performance metrics and DB column lengths; check for truncation and user-friendly error messages.
