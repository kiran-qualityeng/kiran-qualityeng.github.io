Test Cases

EC-BND-01: Login Field Length Boundary

Priority: High
Type: Boundary
Area: Authentication
  1. Preconditions: User signup page available; DB accessible.
  2. Steps: Enter username with 1 char; attempt signup -> observe validation. Repeat with max allowed length (e.g., 255 chars) and with 256 chars.
  3. Expected: 1-char and mid-range input accepted; 255 accepted if within limit; 256 rejected with clear error; no server error or truncation.
  4. Notes: Verify DB stores full string and UI shows full value without overflow.

EC-BND-02: Numeric Field Upper Bound (Quantity)

Priority: Medium
Type: Boundary
Area: Orders
  1. Preconditions: Order creation UI and API available.
  2. Steps: Submit order with quantity 0, 1, large number (10,000), and max-int+1; observe response.
  3. Expected: 0 rejected if not allowed; normal range accepted; server rejects overflow with 4xx and informative message; no DB corruption.

EC-IV-01: Input Validation – Special Characters

Priority: High
Type: Validation
Area: User Input
  1. Preconditions: Form accepts free-text inputs (name, comments).
  2. Steps: Submit entries with <, >, ', ", SQL meta-characters, very long multibyte strings.
  3. Expected: UI sanitizes or rejects dangerous characters; backend returns validation errors for prohibited inputs; no XSS or SQL injection possible.
  4. Verification: Inspect network payloads and stored values to ensure encoded or rejected characters.

ST-ORD-01: Order Lifecycle – Pending → Confirmed → Shipped → Delivered

Priority: High
Type: Workflow
  1. Preconditions: Order API with state transitions and webhook/notification system active.
  2. Steps: Create order (Pending), trigger payment to move to Confirmed, simulate fulfillment -> move to Shipped, then mark Delivered.
  3. Expected: Each state transition only allowed from previous valid state, timestamps and audit trail captured, notifications fired, and UI reflects state.

ST-ACC-01: User Account Activation & Password Reset Flow

Priority: Medium
Type: State & Security
  1. Preconditions: New user signup and password reset endpoints available.
  2. Steps: Register new user -> verify activation email link expired vs valid; request password reset -> use token; attempt reuse of token.
  3. Expected: Activation works only once; expired tokens rejected; reused tokens invalidated; audit logs include IP & timestamp.

DI-FU-01: File Upload – Size & Type Validation

Priority: High
Type: Functional
  1. Preconditions: Upload endpoint with file-type whitelist and size limit (e.g., 10MB).
  2. Steps: Upload allowed type within limit; upload disallowed type; upload file slightly over limit; upload corrupted file stream.
  3. Expected: Allowed files succeed; disallowed types rejected with proper code; oversized rejects with message; corrupted stream handled gracefully without server crash.

DI-DC-01: Data Corruption – Simulate Partial Writes

Priority: Medium
Type: Resilience
  1. Preconditions: Backend storage and atomic write operations available.
  2. Steps: Simulate network interruption during record write; read record after interruption.
  3. Expected: System either rolls back or marks record as inconsistent for remediation; no silent partial writes that break referential integrity.

SEC-AUTH-01: Authorization – Role-Based Access Control

Priority: High
Type: Security
  1. Preconditions: Users with roles (admin, manager, user) exist.
  2. Steps: Attempt admin-only action as regular user via UI and direct API call; attempt manager-level action as user.
  3. Expected: Unauthorized attempts receive 403; no privilege escalation; logs record attempts.

SEC-INS-01: Input Sanitization – Prevent XSS & Injection

Priority: High
Type: Security
  1. Preconditions: Inputs stored and rendered in UI (comments, product names).
  2. Steps: Submit payloads containing script tags, event handlers, JSON with injection patterns.
  3. Expected: UI escapes content or rejects unsafe input; no script execution; server uses parameterized queries.

INT-FS-01: Downstream Service Failure – Retry & Circuit Breaker

Priority: High
Type: Integration
  1. Preconditions: App depends on external pricing API with retry logic and circuit breaker.
  2. Steps: Simulate 500/504 from service; verify retry behavior, circuit open after threshold, fallback values used if configured.
  3. Expected: Retries attempted up to limit, circuit opens and prevents cascading retries, user sees meaningful message or fallback content.

INT-FS-02: Message Queue Processing Failure

Priority: Medium
Type: Integration/Recovery
  1. Preconditions: System consumes messages from queue and performs idempotent operations.
  2. Steps: Force handler to throw on first attempt; ensure message moves to DLQ after N retries; reprocess DLQ message.
  3. Expected: Failed message does not block queue; DLQ contains diagnostics; reprocessing succeeds or is remedied with repeatable outcome.
