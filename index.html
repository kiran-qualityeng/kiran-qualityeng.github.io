Input Validation — Robust Sanitization & Length Handling (EC-IV)

Priority: Medium / High
Focus: XSS, SQLi, Length Limits
  1. EC-IV-001 — Special Character Handling
    1. Preconditions: Forms and APIs accepting user text fields are available.
    2. Steps:
      1. Submit input: John<script>alert('xss')</script>Doe in name/comment fields.
      2. Submit input: test'; DROP TABLE users;-- via form and direct API payload.
      3. Submit a normal input string and compare behaviour.
    3. Expected: Inputs are sanitized or rejected; no script execution in UI; server rejects/validates and encodes on output; logs show validation errors (not stack traces).
    4. Verification: Inspect DOM for injected scripts, review network payloads, and confirm DB stored values are encoded or rejected.
  2. EC-IV-002 — Extremely Long Inputs
    1. Preconditions: UI and API have defined max length policies.
    2. Steps:
      1. Enter 500 characters in name field; 10,000 in description; 1000 in search; submit forms and API calls.
      2. Measure response latency and observe behavior with very long Unicode/multibyte strings.
    3. Expected: UI truncates or prompts error; API validates payload length and returns 4xx with clear message; no backend slowdowns or crashes; DB stores allowed lengths only.
    4. Verification: Monitor performance metrics and DB column lengths; check for truncation and user-friendly error messages.

Order Workflow — Prevent Invalid Transitions & Cancellation Handling (ST-OW)

Priority: High
Focus: Workflow correctness & audit trail
  1. ST-OW-001 — Prevent Invalid Status Changes
    1. Preconditions: Order management UI/API with status enum (PENDING, CONFIRMED, SHIPPED, DELIVERED).
    2. Steps:
      1. Locate order in PENDING state.
      2. Attempt to change directly to SHIPPED or DELIVERED via UI and direct API.
      3. Perform valid flow PENDING → CONFIRMED → SHIPPED → DELIVERED.
    3. Expected: Invalid transitions blocked with clear error (e.g., "Cannot ship order that is not confirmed"); valid transitions succeed and generate audit entries (who/when/from→to).
  2. ST-OW-002 — Cancellation Behavior Across States
    1. Preconditions: Cancellation operation exists and inventory service connected.
    2. Steps:
      1. Cancel order in PENDING — expect success and inventory release.
      2. Cancel in CONFIRMED — expect success, refund flow initiated and inventory release.
      3. Attempt cancel in SHIPPED — expect rejection with message "Cannot cancel shipped order".
      4. Attempt cancel in DELIVERED — expect rejection with message "Cannot cancel delivered order".
    3. Expected: Correct outcomes, inventory updates, refund or hold behavior, and clear user messages. Audit logs updated.

User Account Lifecycle — Activation, Suspension & Deletion (ST-UA)

Priority: High
Focus: Auth & state enforcement
  1. ST-UA-001 — Account Status Flow
    1. Preconditions: Signup, verification, suspend/delete endpoints exist.
    2. Steps:
      1. Register new account → UNVERIFIED.
      2. Attempt login as UNVERIFIED (should fail with "Verify your email").
      3. Verify email → ACTIVE, login should succeed.
      4. Suspend account → SUSPENDED; login should fail with "Account suspended".
      5. Delete account → DELETED; login attempts should return "Account not found".
    3. Expected: Only valid states allow actions; tokens invalidated on suspension/deletion; audit trail records IP & timestamps.

File Upload & Data Integrity — Type Detection & Corruption Handling (DI-FU)

Priority: High
Focus: Security & resilience
  1. DI-FU-001 — File Type Spoofing
    1. Preconditions: Upload endpoint with type/size rules and file signature check.
    2. Steps:
      1. Rename a .txt to .jpg and upload.
      2. Rename .exe to .pdf and upload.
      3. Upload valid .jpg/.pdf file.
      4. Upload file with double extension like file.jpg.exe.
    3. Expected: Spoofed and double-extension files rejected with "Invalid file type"; valid files accepted; server checks magic bytes/signature, not extension alone.
  2. DI-FU-002 — Corrupted File Handling
    1. Preconditions: Upload service and scanning pipeline in place.
    2. Steps:
      1. Upload JPEG with corrupted header.
      2. Upload PDF missing EOF marker.
      3. Upload empty (0 bytes) file.
      4. Upload very large file exceeding size limit.
    3. Expected: Clear, specific error messages; no server crashes; files rejected or quarantined; logs retained for investigation; size limits enforced.

Authorization & Access Control — Prevent Data Leaks & API Abuse (SEC-AUTH)

Priority: Critical
Focus: Horizontal & API security
  1. SEC-AUTH-001 — Horizontal Privilege Escalation
    1. Preconditions: Multiple user accounts and order/profile endpoints exist.
    2. Steps:
      1. Login as User A (ID 123). Note URL /orders/123.
      2. Change URL to /orders/124 and attempt to view.
      3. Repeat for /profile/124 and /documents/124.
    3. Expected: 403 Forbidden for other users' resources; no data leaked; server-side checks validate owner; logs capture denied attempts.
  2. SEC-AUTH-002 — API Authentication & Token Validation
    1. Preconditions: Protected APIs and token handling implemented.
    2. Steps:
      1. Call protected API without login (expect 401).
      2. Login as normal user and call admin API (expect 403).
      3. Use expired token to call protected endpoints (expect 401).
    3. Expected: Proper 401/403 responses, minimal sensitive info in errors, token expiry and revocation enforced server-side.

Integration Resilience — Payment & External Service Failures (INT-FS)

Priority: High
Focus: Retry, fallback & recovery
  1. INT-FS-001 — Payment Gateway Timeout & Retry
    1. Preconditions: Checkout flow and payment gateway integration configured with retry/fallback logic.
    2. Steps:
      1. Initiate checkout and simulate payment gateway timeout.
      2. Check order status in DB and user-facing message.
      3. Retry payment and confirm no duplicate charges.
    3. Expected: Order set to PAYMENT_PENDING (not stuck in PROCESSING); user sees clear retry message; idempotent retry prevents duplicate charges; inventory not permanently reserved incorrectly.
  2. INT-FS-002 — External Service Degradation
    1. Preconditions: Multiple external services (email, SMS, address validation) integrated.
    2. Steps:
      1. Disable email service and register user; observe behavior.
      2. Disable SMS and request password reset; observe fallback strategies.
      3. Disable address validation and submit address.
    3. Expected: Application continues with degraded features, queues messages for retry, shows clear explanatory messages, and no data loss.
